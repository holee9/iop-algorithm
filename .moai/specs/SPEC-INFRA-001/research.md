# SPEC-INFRA-001 연구 노트 — T0 프레임워크 스캐폴드

XDET 영상처리 SW P1의 최우선 작업 T0(프레임워크)에 대한 아키텍처 분석. 본 문서는 구현 이전 단계(Plan Phase 1B)의 근거 분석이며, 코드를 포함하지 않는다.

## 1. 목적과 범위

T0는 다른 모든 작업(T1~T10)의 전제 조건이다. 처리 알고리즘을 하나도 구현하지 않으면서, 13개 파이프라인 모듈과 지표 엔진이 준수해야 할 **공통 계약과 검증 기계장치(machinery)**를 확립한다.

산출 대상(task 지시):
- repo 스캐폴드 (`common/ modules/ pipeline/ metrics/ tests/ data/`)
- `XFrame` 공통 데이터 컨테이너
- `CalibSet` 공통 스키마
- 오케스트레이터(파이프라인 정의)
- 모듈 단위 시험 harness
- 의존 방향 정적 검사기
- pytest/CI 골격

DoD: **XDET-TC-000 통과** — 모듈 fixture 입출력 일치 + 시그니처·의존 방향 정적 검사, 계약 위반 0건.

## 3. 아키텍처 분석 — 5개 하위 시스템

T0 산출물을 5개의 요구 모듈로 분해한다(요구 모듈 ≤5 제약 충족). 각 모듈은 특정 SWR-000 조항 집합에 대응한다.

### 3.1 데이터 계약 — XFrame + CalibSet (SWR-000-6, -10, -3, -4)
- `XFrame`은 모든 모듈의 **유일한 입출력 단위**. 4개 구성요소(pixel float32 / 마스크 스택 / 노이즈 모델 α,σ / 처리 이력 체인)를 담는 불변(immutable) 컨테이너로 설계한다.
- 마스크 스택은 defect/포화/보간을 비트플래그로 표현 — 향후 T2(defect)·T3(포화)가 소비할 슬롯을 T0에서 예약한다.
- `CalibSet`은 단일 스키마: 패널 ID·해상도·유효기간·종류(offset/gain/defect/lag IRF/노이즈 모델/scatter 커널)·데이터·생성 이력. 스키마 검증기(schema validator)를 함께 둔다.
- 처리 이력 체인은 SWR-000-4의 메타(모듈 버전·파라미터 해시·캘리브 파일 ID)를 누적 — IEC 62304 추적성의 실체.
- SWR-000-3의 "검증 모드 중간 산출 보존"은 컨테이너 수준 옵션 플래그로 설계.

### 3.2 모듈 계약 + harness (SWR-000-7, -11)
- 전 모듈 단일 시그니처 `process(XFrame, CalibSet, Params) -> XFrame`, 순수함수형. T0는 이 시그니처를 강제하는 **추상 계약(프로토콜/베이스)**과, 이를 준수하는 **레퍼런스 passthrough 모듈**(항등 변환 예시)을 제공하여 harness와 정적검사를 자기 검증한다.
- lag 등 내부 상태 모듈은 상태를 컨테이너로 직렬화 가능해야 함 — T0는 이 직렬화 계약 자리만 마련(구현은 T4).
- harness: 합성 입력 fixture + 기대 출력 fixture를 로드해 `process()` 결과와 XFrame 단위 비교. 이것이 TC-000의 "fixture 입출력 일치" 판정 엔진.

### 3.3 오케스트레이터 + 순서 고정 (SWR-000-8, -2, -5)
- 오케스트레이터(파이프라인 정의 파일)만이 모듈 순서·조합을 결정. 모듈은 서로를 호출하지 않는다.
- 고정 순서 offset → gain → defect → lag → line noise → (포화/기하) → post를 파이프라인 정의로 표현. T0에는 실제 모듈이 없으므로 **레퍼런스 passthrough 모듈들의 순서 조립**으로 골격을 실증한다.
- SWR-000-5 예외 정책: CalibSet 부재/불일치(해상도·패널 ID) 시 처리 거부 + 명시 오류. 무단 기본값 대체 금지 — 오케스트레이터 진입 시 검증 게이트로 구현.

### 3.4 의존 방향 정적 검사 + 공용 컴포넌트 스캐폴드 (SWR-000-8, -9)
- 정적 검사기: import/호출 그래프를 분석해 (a) 모듈→모듈 직접 호출, (b) common→module 역방향, (c) 모듈 간 수평 의존을 검출하면 실패. 이것이 TC-000의 "의존 방향 정적 검사" 판정 엔진.
- 공용 컴포넌트 5종(pyramid / histogram·FOV / fft_psd / robust_stats / mask_ops)은 `common/`에 **디렉터리·인터페이스 스텁만** 배치(중복 방지 계약 확립). 실제 알고리즘은 소비 WP(T1·T5·T6·T7 등)에서 채운다.

### 3.5 CI 골격 + 정밀도/동일성 훅 (SWR-000-1, -11, -12, TC-000, TC-021 훅)
- pytest/CI 골격: TC-000~021을 pytest에 1:1 등록하는 구조. T0는 **TC-000을 실동작**시키고 나머지는 skeleton(placeholder/skip)로 등록.
- 커밋 트리거 → 모듈 단위 게이트(TC-000) → 계약 위반 시 머지 차단.
- SWR-000-1: float32 기본 + float64 병행 산출 경로를 XFrame 연산 계층에 설계 → 향후 수치 안정성 대조의 근거.
- SWR-000-12 구현 교체 계약: 골든/최적화/FPGA 등가성 비교를 위한 **diff 유틸 훅**을 배치(TC-021 계열이 꽂힐 자리). 수치 임계(±1 LSB / bit-동일)는 P2로 이연 — T0는 구조만.

## 4. 제약 조건 (Constraints)

- **[HARD] 사이드채널 금지** (SWR-000-6): 모듈 간 데이터 전달은 오직 XFrame으로만. 전역 상태·부가 반환값·파일 경유 우회 모두 금지.
- **[HARD] 단일 시그니처·순수함수** (SWR-000-7): 부작용 없는 `process(XFrame, CalibSet, Params) -> XFrame`. 입력 XFrame 불변(SWR-000-3).
- **[HARD] 의존 방향 단방향** (SWR-000-8): module → common 만 허용. 정적 검사가 CI에서 강제.
- **[HARD] 공용 컴포넌트 1회 구현** (SWR-000-9): 5종 중복 금지.
- **[HARD] CalibSet 부재 시 기본값 대체 금지** (SWR-000-5): 명시 오류로 처리 거부.
- **[HARD] 파이프라인 순서 임의 변경 금지** (SWR-000-2).
- **정확도 단일 목표**: 속도 최적화 금지(tech.md), numpy/scipy float 골든 모델. T0도 최적화보다 계약 명료성 우선.
- **파라미터 외부화**: 조정 가능 수치는 Params/CalibSet로 외부화, 하드코딩 금지(TBD-[B] 9건 / TBD-[T] 11건 / [P] 등급). T0는 Params 주입 경로를 계약에 포함.

## 6. 권고 (Recommendations)

1. **레퍼런스 passthrough 모듈을 T0 산출물에 포함** — TC-000을 실동작시키는 유일한 방법. 이 모듈이 계약 준수 예시이자 harness/정적검사의 자기 검증 대상.
2. **XFrame을 float32 pixel + float64 병행 산출 옵션으로 설계** (SWR-000-1) — 향후 수치 안정성 대조·TC-021 diff의 토대. 이를 T0에서 놓치면 T10에서 광범위 재작업.
3. **정적 검사기를 CI 필수 게이트로 1급 배치** — 사후 도입 시 이미 축적된 위반을 되돌리기 어렵다. T0부터 "위반 0건" 불변식 유지.
4. **CalibSet 스키마 검증기 + SWR-000-5 거부 게이트를 오케스트레이터 진입점에 배치** — 무단 기본값 대체를 구조적으로 봉쇄.
5. **공용 컴포넌트는 스텁으로만, 시그니처 확정은 첫 소비 WP로 이연** — 조기 과설계 회피(Enforce Simplicity).
6. **TC-021 수치 임계는 명시적으로 P2 이연, T0는 diff 유틸 훅만** — 범위 규율 유지. spec.md의 Exclusions에 반드시 명기.
7. **처리 이력 해시 체인을 결정론적으로 설계** — 동일 입력·파라미터에 대해 재현 가능한 해시. 구현 교체 계약(SWR-000-12) 검증의 전제.
